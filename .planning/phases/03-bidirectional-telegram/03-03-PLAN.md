    Two changes to server.ts in the `main()` function:

    **Change 1 — Start polling on boot:**

    The current server.ts has this block in main():
    ```typescript
    let scheduler: ScanScheduler | null = null;
    if (config.ebay && config.telegram) {
      const alerts = new TelegramAlerts(config, storage);
      scheduler = new ScanScheduler(
        config,

      scheduler.start();
      console.log('[Server] Background scheduler started');
    } else {
      console.log('[Server] Scheduler not started — eBay or Telegram credentials missing');
    }
    ```

    The problem: `alerts` is scoped inside the `if` block and is not accessible to the shutdown handler defined later. Refactor to hoist `alerts` outside the if block, so the shutdown handler can call stopPolling():

    ```typescript
    let scheduler: ScanScheduler | null = null;
    let alerts: TelegramAlerts | null = null;
    if (config.ebay && config.telegram) {
      alerts = new TelegramAlerts(config, storage);
      scheduler = new ScanScheduler(
        config,
        watchlist,
        scanner!,
        priceEngine,
        dealScorer,
        alerts,
        storage,
      );
      scheduler.start();
      console.log('[Server] Background scheduler started');
      alerts.startPolling();
      console.log('[Server] Telegram polling started');
    } else {
      console.log('[Server] Scheduler not started — eBay or Telegram credentials missing');
    }
    ```

    **Change 2 — Stop polling in shutdown handler:**

    The current shutdown handler:
    ```typescript
    const shutdown = () => {
      console.log('[Server] Shutting down...');
      scheduler?.stop();
      server.close(() => process.exit(0));
    };
    ```

    Update it to await stopPolling() before closing the server. Since `stopPolling()` is async, make the shutdown handler async:
    ```typescript
    const shutdown = () => {
      console.log('[Server] Shutting down...');
      scheduler?.stop();
      void alerts?.stopPolling().then(() => {
        server.close(() => process.exit(0));
      });
    };
    ```

    Alternatively, if a simpler form is preferred (still correct — server.close fires after in-flight requests finish, polling stop is best-effort):
    ```typescript
    const shutdown = () => {
      console.log('[Server] Shutting down...');
      scheduler?.stop();
      void alerts?.stopPolling();
      server.close(() => process.exit(0));
    };
    ```

    Use the simpler form (void alerts?.stopPolling() then server.close). stopPolling() completing before process.exit is best-effort; the process manager handles unclean exits. Chaining via .then() creates ordering but is not required for correctness here.

    **TypeScript note:** `alerts` is typed as `TelegramAlerts | null`. The optional chaining `alerts?.startPolling()` and `alerts?.stopPolling()` handle the null case. The `void` prefix on the stopPolling call silences the unhandled-promise lint rule.

    No other changes to server.ts.
