## Directory Purposes

**src/:**
- Purpose: All source code (TypeScript)
- Contains: Modules organized by feature area
- Key files: Entry points (`cli.ts`, `server.ts`, `index.ts`), barrel exports

**src/types/:**
- Purpose: Centralized type definitions for the entire application
- Contains: All interfaces and types (ResolvedCard, WatchlistEntry, FairMarketValue, EbayListing, ScoredDeal, etc.)
- Key files: `index.ts` — 200+ lines of exported types and one const (DEFAULT_SCHEDULER_CONFIG)

**src/card-resolver/:**
- Purpose: Card disambiguation and resolution from natural language
- Contains: CardResolver class for parsing card descriptions via external API
- Key files: `resolver.ts` — ~140 lines, integrates PokemonPriceTracker parse-title API

**src/pricing/:**
- Purpose: Fair market value lookups and caching
- Contains: PriceEngine class with in-memory 30-minute TTL cache
- Key files: `engine.ts` — ~100 lines, fetches from PokemonPriceTracker PSA pricing endpoint

**src/scanner/:**
- Purpose: eBay marketplace scanning and deal analysis
- Contains:
  - `ebay.ts`: Authenticates with eBay API (OAuth2 app ID + cert), builds search queries, fetches listings
  - `deal-scorer.ts`: Scores listings using weighted algorithm (60% price vs FMV, 15% seller reputation, 10% listing type, 15% population rarity)
- Key files:
  - `ebay.ts` — ~200 lines
  - `deal-scorer.ts` — ~150 lines
  - `deal-scorer.test.ts` — 120 lines with 8 test cases

**src/watchlist/:**
- Purpose: User watchlist management and persistent storage
- Contains:
  - `manager.ts`: CRUD operations, user-indexed lookups via `StorageAdapter` abstraction
  - `storage-json.ts`: JSON file implementation of StorageAdapter
  - Tests verify multi-user isolation, activation status, scan tracking
- Key files:
  - `manager.ts` — ~100 lines
  - `storage-json.ts` — ~65 lines
  - `manager.test.ts` — 123 lines with 6 test cases

**src/scheduler/:**
- Purpose: Automated background scanning and alerting on intervals
- Contains: ScanScheduler class orchestrating watchlist iteration, rate limiting, deal scoring, alert sending
- Key files: `scan-scheduler.ts` — ~200 lines, state machine with start/stop/runOnce

**src/alerts/:**
- Purpose: External alert notification channels
- Contains: TelegramAlerts class formatting deals as HTML messages and posting via Telegram Bot API
- Key files: `telegram.ts` — 123 lines, with deduplication by item ID

**dist/:**
- Purpose: Compiled JavaScript output from TypeScript compiler
- Generated: By `npm run build` (runs `tsc`)
- Committed: No (in .gitignore)

**data/:**
- Purpose: Runtime persistent storage directory
- Contains: `gacha-agent-data.json` — JSON object with watchlist entries and agent records
- Generated: On first write by JsonStorageAdapter
- Committed: No (in .gitignore)

## Key File Locations

**Entry Points:**
- `src/cli.ts`: CLI command handler — user runs commands like `resolve`, `price`, `watch`, `scan`, `run`
- `src/server.ts`: HTTP API server — exposes REST endpoints, handles agent registration and authentication
- `src/index.ts`: Barrel export file for library use (re-exports all public types and classes)

**Configuration:**
- `.env.example`: Template showing required environment variables (POKEMON_PRICE_TRACKER_API_KEY, EBAY_APP_ID, TELEGRAM_BOT_TOKEN, etc.)
- `tsconfig.json`: TypeScript compiler settings (ES2022 target, strict mode, source maps, declaration files)
- `package.json`: Dependencies (dotenv, pokemon-tcg-sdk-typescript), dev dependencies (tsx, typescript, vitest)
- `vitest.config.ts`: Test runner configured to find `src/**/*.test.ts` files

**Core Logic:**
- `src/card-resolver/resolver.ts`: Card disambiguation against Pokemon TCG database
- `src/pricing/engine.ts`: FMV calculation and caching
- `src/scanner/ebay.ts`: eBay API integration and listing fetching
- `src/scanner/deal-scorer.ts`: Weighted deal scoring algorithm
- `src/watchlist/manager.ts`: User watchlist persistence and lookups
- `src/scheduler/scan-scheduler.ts`: Automated scanning orchestration
- `src/alerts/telegram.ts`: Formatted alert message delivery

**Testing:**
- `src/scanner/deal-scorer.test.ts`: 8 test cases covering scoring logic, seller reputation, listing type preference
- `src/watchlist/manager.test.ts`: 6 test cases covering CRUD, user isolation, activity status, scan tracking

**Type Definitions:**
- `src/types/index.ts`: Single source of truth for all domain types (30+ exported interfaces and types)

## Naming Conventions

**Files:**
- Classes: PascalCase match class name (e.g., `CardResolver` → `resolver.ts`, `WatchlistManager` → `manager.ts`)
- Test files: Same name as implementation with `.test.ts` suffix (e.g., `deal-scorer.test.ts`)
- Directories: kebab-case (e.g., `card-resolver/`, `scan-scheduler/`)
- Barrel exports: `index.ts` at module root

**Directories:**
- Feature modules: Single-word or hyphenated (scanner, alerts, pricing, card-resolver, watchlist, scheduler)
- One class per feature file (CardResolver in resolver.ts, PriceEngine in engine.ts)
- Related utilities grouped (deal-scorer and ebay both in scanner/)

**Exports:**
- `src/index.ts` re-exports all public types and classes
- Each module exports one main class per file (named export, no default exports)
- Types all exported from `src/types/index.ts`

## Where to Add New Code

**New Feature (e.g., new marketplace like TCGPlayer):**
- Primary code: `src/scanner/tcgplayer.ts` — new scanner class alongside `ebay.ts`
- Types: Add interfaces to `src/types/index.ts` if needed (TCGListing, etc.)
- Tests: `src/scanner/tcgplayer.test.ts` alongside existing deal-scorer tests
- Integration: Update `src/cli.ts` and `src/server.ts` to instantiate new scanner for commands
- Export: Add to `src/index.ts` barrel export

**New Alert Channel (e.g., Discord):**
- Implementation: `src/alerts/discord.ts` alongside `telegram.ts`
- Pattern: Implement same methods as TelegramAlerts (sendDealAlert, sendText, clearDedupeCache)
- Types: Add to AlertChannel union in `src/types/index.ts` (currently: 'telegram' | 'webhook')
- Integration: Update WatchlistEntry.alertChannels to include new channel
- Export: Add to `src/index.ts`

**New Utility/Helper:**
- Location: Create at module level within relevant feature area
  - Price calculation helpers: `src/pricing/utils.ts`
  - Search query builders: `src/scanner/utils.ts`
  - Formatting helpers: `src/alerts/format.ts`
- Pattern: Export pure functions with no side effects
- No new top-level directory unless it represents a major new feature area

**New Command in CLI:**
- Primary code: Add case in switch statement in `src/cli.ts` around line 76
- Pattern: Each command follows: load config → instantiate services → call service methods → format output
- Placement: After existing cases (resolve, price, watch, list, scan, run)
- Usage string: Add to `printHelp()` function for documentation

**New Endpoint in Server:**
- Primary code: Add route handler in request handler function in `src/server.ts` around line 170
- Pattern: Check method + path → parse body if POST → authenticate → call services → return JSON
- Authentication: Most endpoints require `Bearer` token; register is public
- Placement: After existing endpoints in logical grouping (agents, resolve, price, watch, scan, etc.)

**New Storage Backend (e.g., Supabase):**
- Implementation: `src/watchlist/storage-supabase.ts`
- Pattern: Implement `StorageAdapter` interface from `src/types/index.ts`
- Methods: async get(), async set(), async delete(), async list(prefix)
- Integration: CLI and server use `JsonStorageAdapter` by default; configurable via env var
- Export: Add to `src/index.ts`

## Special Directories

**dist/:**
- Purpose: Compiled JavaScript output
- Generated: Yes (by `npm run build`)
- Committed: No (.gitignore)
- Contents: mirror of src/ structure with .js files and .d.ts type declarations

**data/:**
- Purpose: Runtime persistent storage
- Generated: Yes (created on first storage write)
- Committed: No (.gitignore)
- File: `gacha-agent-data.json` — JSON object with keys like `watchlist:{id}`, `agents`, `heartbeat:{id}`

**.planning/:**
- Purpose: GSD (Get Stuff Done) agent planning documents
- Generated: Yes (by GSD orchestrator)
- Committed: Yes
- Contents: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, CONCERNS.md

## Import Paths & Aliases

**No path aliases** are configured in `tsconfig.json`. All imports use relative paths from file location.

**Pattern:**
- Same directory: `import { CardResolver } from './resolver.js'`
- Parent directory: `import { PriceEngine } from '../pricing/engine.js'`
- Root level: `import type { ResolvedCard } from '../types/index.js'`

**File extensions:**
- All imports end with `.js` (for ES modules compatibility even in TypeScript)
- tsconfig uses `moduleResolution: "NodeNext"` to support this pattern
